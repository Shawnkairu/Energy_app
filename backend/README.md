# e.mappa Backend

FastAPI backend. Exposes the camelCase API contracts used by `@emappa/api-client`; keeps business calculations in organized Python services that mirror `packages/shared`.

## Local dev (recommended: `uv`)

System Python from Homebrew on macOS has a known `pyexpat` linkage bug (broken `_XML_SetAllocTrackerActivationThreshold` symbol). Use [`uv`](https://docs.astral.sh/uv/) to get a self-contained Python that sidesteps it.

```bash
# one-time: install uv
brew install uv     # macOS
# or: curl -LsSf https://astral.sh/uv/install.sh | sh

# one-time: provision Python + venv + deps
cd backend
uv python install 3.12                             # downloads uv-managed CPython
uv venv --python 3.12 .venv                        # creates .venv
uv pip install --python .venv/bin/python -r requirements.lock   # install from lockfile (reproducible)

# daily
.venv/bin/uvicorn app.main:app --reload            # boot backend on :8000
.venv/bin/alembic upgrade head                     # apply migrations
.venv/bin/python -m pytest tests -q                # run tests
```

Or from repo root:

```bash
npm run dev:backend
npm run migrate
npm run test:backend
```

Health check: `GET http://localhost:8000/health` → `{"status":"ok","db":"ok"}`.

## Dependency management

- `requirements.txt` — top-level dependencies with version *ranges* (`>=`). Edit when adding/removing a package.
- `requirements.lock` — autogenerated by `uv pip compile requirements.txt -o requirements.lock`. Pins every transitive dependency to a specific version. **This is what `uv pip install` reads.** Regenerate (and commit) whenever `requirements.txt` changes:

```bash
cd backend && uv pip compile requirements.txt -o requirements.lock --python .venv/bin/python
```

The lockfile is what makes "works on my machine" the same as "works in CI" the same as "works on a teammate's fresh clone."

## Docker (alternative)

If you'd rather not install uv, the repo has a Dockerfile + `docker-compose.yml` at the root:

```bash
docker compose up --build
```

This bundles Postgres + backend + frontends in one shot.

## Postgres setup

The backend expects a Postgres reachable at `EMAPPA_DATABASE_URL` (see `backend/.env.example` for the default). On macOS:

```bash
brew install postgresql@16
brew services start postgresql@16
createdb emappa
psql -d postgres -c "CREATE ROLE emappa WITH LOGIN PASSWORD 'emappa';"
psql -d postgres -c "GRANT ALL PRIVILEGES ON DATABASE emappa TO emappa;"
```

Then `.venv/bin/alembic upgrade head` from `backend/`.

## Tests

Backend tests live in `backend/tests/`. They use a session-scoped `pytest_configure` hook in `conftest.py` to delete `pilot-*@emappa.test` fixtures at the start of every run, so they're idempotent against a persistent dev DB.

`npm run ci` (from repo root) includes `npm run test:backend` and fails the gate on any backend regression — same gate the JS workspaces use.
